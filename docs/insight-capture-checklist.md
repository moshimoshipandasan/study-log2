# Insight Capture Checklist

学習ログを「なるほど」と読める記録にするための確認リストです。

## 記録すべき流れ

学習者の思考は、次の順番で残します。

1. 最初の見方
2. 問題提起
3. 仮説
4. 根拠確認
5. 思考の変化
6. 得た知見
7. 次に使える判断基準
8. まだ残る問い

## よい知見ログの条件

- 学習者の最初の考えが書かれている。
- その考えのどこがよく、どこが粗かったかが分かれている。
- 見方が変わったきっかけが書かれている。
- 新しく区別できるようになった概念がある。
- 「次は何を確認すればよいか」が判断基準として残っている。
- 未確認のことを断定せず、要確認として残している。

## 弱い記録の例

```md
秀長旧領について調べた。増田長盛が入ったことがわかった。
```

これは事実メモとしては使えますが、学習者の思考が見えません。

## よい記録の例

```md
最初は、秀長旧領が五奉行に分け与えられたため豊臣政権が弱体化した、と考えた。
しかし資料を確認すると、五奉行全員に均等分割されたというより、五奉行の一人である増田長盛が大和郡山20万石に入り、紀伊・和泉の支配も命じられたと整理する方が正確だった。
そのため、秀長旧領の再編は「五奉行への分割」ではなく、「豊臣一門の大領が政権官僚的な管理へ移った」と表現する方が安全だとわかった。
この知見により、秀長・秀保の死は、有能な補佐役を失っただけでなく、五大老に対抗しうる豊臣一門の支柱を失った問題として説明できる。
```

## Codexが補うべき問い

学習者の発言が短い場合、Codexは次の問いを補って整理します。

- 今いちばん知りたいことは何か。
- 答えを見る前に、自分ではどう考えたか。
- 何を見れば、その考えが合っているか確かめられるか。
- 確かめたあと、最初の考えからどこを直すか。
- 今回「なるほど」と思ったことは何か。
- 似た問題を見たら、次は何から確認するか。
- その考えは、何と何を比べているのか。
- その説明は、原因を言いすぎていないか。
- 事実確認が必要な部分はどこか。
- より安全な表現にするとどうなるか。
- 次に同じ型の問題を扱うとき、どんな順番で確認すればよいか。

## Close前の知見チェック

IssueをCloseする前に、次の3文が書けるか確認します。

```text
最初は＿＿＿＿と考えていた。
しかし＿＿＿＿を確認して、＿＿＿＿と考えるようになった。
次に似た問題では、＿＿＿＿を確認する。
```
