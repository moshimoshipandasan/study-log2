# Auto Recording Workflow

会話中の問題提起を、GitHub Issueと学習ログに残すための運用です。

## 目的

- 学習者の疑問、仮説、誤解、調べた根拠、確認問題の結果を流さずに残す。
- 「答えをもらった」ではなく「何を考え、どこが変わったか」を追跡できるようにする。
- Issueを学習単位、`learning-log/` を振り返り単位として使い分ける。
- 学習者が「なるほど」と思えた理由と、次に使える知見を残す。
- 小学生が読み返してわくわくできるように、難しい言葉は「ふしぎ」「予想」「たしかめ」「なるほど」「次のコツ」に置きかえる。

## 記録を始める合図

Codex / Copilotは、次のような発言が出たら記録対象として扱います。

- 「ハーネススタート」
- 「ハーネス開始」
- 「ハーネス終了」
- 「ハーネス完了」
- 「わからない」
- 「なぜ」
- 「どう考えればいい」
- 「この視点を入れたい」
- 「この説明で合っているか」
- 「レポートを作りたい」
- 「間違えた」
- 「確認問題を解きたい」
- 「Issueに記録して」
- 「学習履歴を残して」

`ハーネススタート`、`ハーネス開始`、`学習ハーネスを始めて`、`このテーマでハーネスを使いたい` などは、学習セッション開始の明示的な合図として扱います。
この発言、または開始したいニュアンスが明確な発言が出たら、Codexは教科、テーマ、目的、現在の理解、わからないこと、自分で考えたこと、希望する学習モードを確認し、Issue化または既存Issueへの追記に進みます。
すでに必要な情報が会話に含まれている場合は、不足分だけ確認します。
Issue化または既存Issueへの追記に進む前に、CodexはIssueラベルを整理します。
開始時のラベル整理では、Issueが「何の学習で、どんな種類で、どんな状態か」をあとから見てもわかるようにします。
最低でも `状態:*`、`種類:*`、`公開:*` を確認し、判断できる場合は `教科:*`、`助け:*`、`むずかしさ:*` も付けます。
GitHub Pagesの学習レポートに載せる通常の学習は `公開:のせる` を付け、個人情報、未整理の提出物、公開したくない内容は `公開:のせない` を付けます。
`公開:のせる` と `公開:のせない` は同時に付けません。
学習テーマまたはIssue番号が判断できたら、`study/YYYYMMDD-topic-slug` または `study/YYYYMMDD-issue-N-topic-slug` の形式で学習用ブランチを作成します。
ブランチ作成前には現在のブランチと未コミット変更を確認し、無関係な変更が混ざっている場合は扱いを確認します。

### 開始時ラベル整理

ハーネス開始時は、Issue本文または最初のIssueコメントに `## ラベル整理` を残します。

```md
## ラベル整理

- 教科: `教科:*`
- 種別: `種類:*`
- 状態: `状態:はじめられる`
- 必要な支援: `助け:*`
- 公開可否: `公開:のせる` または `公開:のせない`
```

種別は次の基準で選びます。

- `種類:しくみを知る`: 言葉や仕組みを理解する学習
- `種類:練習する`: 問題を解いて練習する学習
- `種類:まちがい発見`: 間違えた理由を整理する学習
- `種類:ふりかえり`: 前に学んだことをもう一度確認する学習
- `種類:テスト練習`: テストに向けて確認する学習
- `種類:文章を書く`: 文章、作文、レポートを書く学習
- `種類:ふしぎメモ`: 疑問や確かめたいことを出した学習

判断に迷う場合は、いったん安全な仮ラベルを付け、Issueコメントに「仮ラベル」と明記します。

### 開始時一周記録

ハーネス開始時は、最初のIssue作成または最初のIssueコメントで必ず「開始時一周記録」を残します。
これは、開始直後の思考ループ1周目を欠かさず記録するための必須処理です。

開始時一周記録では、同じIssueコメント内に次の3見出しを必ず入れます。

- `## 問題提起`
- `## 学習者の仮説・考え`
- `## 次の確認問題`

この3見出しは、思考ループ判定で `問い`、`仮説`、`確認` として読まれます。
見出し名を言い換えず、短い開始コメントでも省略しません。

学習者の仮説がまだ不足している場合、Codexは勝手に仮説を作りません。
その場合は `## 学習者の仮説・考え` に「未確認。開始直後に本人の予想を確認する。」と書き、`## 次の確認問題` で「まず自分ではどう考えるかを1〜3文で書く」問いを置きます。

開始時一周記録の最小テンプレート:

```md
## 学習開始

ハーネス開始を受けて、このIssueを学習セッションの起点にする。

## ラベル整理

- 教科: `教科:*`
- 種別: `種類:*`
- 状態: `状態:はじめられる`
- 必要な支援: `助け:*`
- 公開可否: `公開:のせる` または `公開:のせない`

## 問題提起

学習者が今回わかりたいこと、解きたい問題、書きたいレポートテーマを書く。

## 学習者の仮説・考え

学習者がすでに書いた予想や考えを書く。
まだ出ていない場合は「未確認。開始直後に本人の予想を確認する。」と書く。

## Codexの整理

最初に分けて考える観点、確認する順番、注意点を短く整理する。

## 次の確認問題

まず自分ではどう考えるかを1〜3文で書く。

## 次回復習

未定。確認問題または最初の説明後に決める。
```

`ハーネス終了`、`ハーネス完了`、`この学習を終わりにしたい`、`Issueを閉じて次に進みたい` などは、現在の学習セッションの終了処理として扱います。
この発言、または終了したいニュアンスが明確な発言が出たら、CodexはClose前チェックを行い、条件を満たす場合はIssueに終了コメントを追記し、必要に応じて `learning-log/` を作成または更新します。
条件を満たさない場合はIssueをCloseせず、不足している記録、確認問題、復習予定、学習者自身の説明など、終了前に必要な最小アクションを提示します。
条件を満たす場合は、学習ブランチ上の変更をコミットまたはPR化し、`main` へ合流してからIssueをCloseします。
`Portfolio` workflow はIssue closeイベントで `main` の内容を使うため、main合流後にIssueをCloseする順序を守ります。
未コミット変更が残っている場合は、`main` に戻る前に止まり、変更の扱いを確認します。

## Issueの選び方

### 既存Issueに追記する場合

次の条件に当てはまる場合は、新しいIssueを作らず、既存Issueにコメントします。

- 同じ学習テーマの続きである。
- 直前のIssueの確認問題、根拠確認、補足論点である。
- レポートや解答の改訂途中である。
- 完了条件がまだ残っている。

### 新しいIssueを作る場合

次の条件に当てはまる場合は、新しいIssueを作成します。

- 教科またはテーマが変わった。
- 目的や完了条件が前のIssueと違う。
- 別の間違いレビューとして扱う方がよい。
- 前のIssueをCloseした後の新しい学習課題である。

## Issueコメントの形式

Issueコメントは、必要に応じて次の形で残します。

```md
## 問題提起

## 学習者の仮説・考え

## 思考の変化

## なるほどポイント

## 得た知見

## 次に使える判断基準

## Codexの整理

## 根拠確認

## 誤解しやすい点

## 次の確認問題

## 次回復習
```

すべての見出しを毎回使う必要はありません。
ただし、ハーネス開始時だけは例外です。
開始時一周記録では `## 問題提起`、`## 学習者の仮説・考え`、`## 次の確認問題` を同じIssueコメント内に必ず入れます。
ただし、学習が進んだ場合は「思考の変化」「得た知見」「次に使える判断基準」のどれかを必ず残します。

## 知見化の観点

Issueコメントや学習ログを書くときは、次の問いで学習者の思考を整理します。

- 学習者は最初に何を問題だと思ったか。
- 最初の仮説はどこまで正しかったか。
- どの事実、資料、反例、比較で見方が変わったか。
- 何を区別できるようになったか。
- どの表現を避け、どの表現なら安全だとわかったか。
- 次に似たテーマを扱うとき、どの順番で確認すればよいか。
- まだ調べるべき根拠は何か。

### 知見ログの書き方

学習者の思考は、できるだけ次の型で残します。

```md
## 思考の変化

最初はAと考えた。理由はBだった。
しかしCを確認したことで、Aだけでは説明が粗いとわかった。
その結果、Dと考える方が安全だと整理した。

## 得た知見

- AとBは分けて考える必要がある。
- 断定するときは、Cの根拠が必要である。
- 今回のような問題では、D、E、Fの順に確認するとよい。
```

「なるほどポイント」は、学習者が納得した理由を短く書きます。

```md
## なるほどポイント

五奉行は政務を担う存在であり、五大老のような大大名と同じ軍事的基盤を持つわけではない。
そのため、秀長旧領の再編は単なる土地の移動ではなく、豊臣一門の対抗力が弱まる問題として見られる。
```

## 学習ログにするタイミング

Issueコメントは会話中の記録、`learning-log/` は学習の区切りの振り返りとして使います。

次のどれかを満たしたら、Codexは `learning-log/YYYY-MM-DD-topic.md` を作成または更新します。

- レポート本文、解答案、説明文などの成果物ができた。
- 誤解の原因が整理できた。
- 確認問題を1問以上解いた。
- 次回復習日が決まった。
- IssueをCloseできる状態になった。

## Close前チェック

IssueをCloseする前に、次を確認します。

- 学習テーマと目的が明確である。
- 最初にわからなかったことが記録されている。
- 学習者の仮説または自分で考えたことが記録されている。
- Codexの説明だけでなく、学習者の言葉での説明がある。
- 学習者の思考がどう変わったかが記録されている。
- 学習者が得た知見または次に使える判断基準が記録されている。
- 類題または確認問題を1問以上解いている。
- 間違いの原因、または誤解しやすい点が記録されている。
- 次回復習する内容または日付が決まっている。

条件が満たされない場合、IssueはCloseせず、未完了タスクをコメントします。

条件が満たされた場合は、IssueをCloseする前にまなびレポートHTMLを生成します。
Issue番号がわかる場合は、次を実行します。

```powershell
npm run harness:end-report -- --issue <Issue番号>
```

ローカルの学習ログから生成する場合は、次を実行します。

```powershell
npm run harness:end-report -- --source learning-log/YYYY-MM-DD-topic.md
```

生成先は `public/thinking-depth.html` です。
このHTMLは、その回の学習について「ふしぎ」「予想」「たしかめ」「考え直し」「なるほど」「次のコツ」を小学生向けの言葉で表示します。
点数表ではなく、Issueコメントや学習ログから思考の道すじが読めるかを確認するための終了時レポートです。
Markdown版も必要な場合は、`--markdown-out public/thinking-depth.md` を付けます。

GitHub Pagesで公開する場合は、終了コメントと学習ログを整え、学習ブランチを `main` へ合流してからIssueをCloseします。
Issue closeイベントで `Portfolio` workflow が自動実行され、全体ダッシュボードと終了時レポートがPagesへデプロイされます。
このワークフローは `main` の最新状態から実行されるため、IssueをCloseする前にmain合流が完了している必要があります。
PR本文に `Closes #<Issue番号>` を入れる場合は、PR mergeによる自動Closeを使うと、merge後にデプロイが始まるため順序が崩れにくくなります。

workflowが起動しない場合だけ、手動フォールバックとして次を実行します。

```powershell
gh workflow run portfolio.yml -f issue_number=<Issue番号>
gh run watch
```

`Portfolio` workflowは、全体ダッシュボードの `index.html` に加えて、最新レポートの `thinking-depth.html` とIssue別固定レポートの `thinking-depth/issue-<Issue番号>.html` をPages artifactに含めます。
同時に、Pagesの権限や設定の問題でHTML公開が失敗した場合に備えて、`portfolio.md`、`thinking-depth.md`、`thinking-depth/issue-<Issue番号>.md` を `portfolio-markdown` というActions artifactとしてアップロードします。
Pages公開に失敗した場合は、GitHubのActions画面で該当runを開き、Artifactsから `portfolio-markdown` をダウンロードしてMarkdown版を確認します。

## ハーネス終了時のコメント形式

`ハーネス終了` で学習を閉じるときは、Issueコメントに必要に応じて次を残します。

```md
## 学習終了まとめ

## 今日できるようになったこと

## 最初の見方

## 思考の変化

## 得た知見

## 次に使える判断基準

## 確認問題の結果

## 間違いの原因・誤解しやすい点

## 次回復習

## 次の課題候補
```

IssueをCloseできない場合は、次のように未完了タスクを残します。

```md
## ハーネス終了前の未完了タスク

- [ ] 確認問題を1問解く
- [ ] 自分の言葉で説明を書く
- [ ] 次回復習日を決める
```

## ブランチ運用

学習セッションごとに、日付と内容がわかるブランチを使います。

```text
study/YYYYMMDD-topic-slug
study/YYYYMMDD-issue-N-topic-slug
```

例:

```text
study/20260512-linear-function
study/20260512-issue-12-english-reading
```

開始時の基本手順:

```powershell
git status --short --branch
git switch main
git pull --ff-only
git switch -c study/YYYYMMDD-topic-slug
```

終了時の基本手順:

```powershell
git status --short --branch
git switch main
git pull --ff-only
```

未コミット変更が残っている場合は、コミット、PR作成、学習ログ追記などを行ってから `main` に戻ります。
無関係な変更が混ざっている場合は、勝手に戻さず、扱いを確認します。

## GitHub CLIでの記録例

既存Issueに追記する例:

```powershell
$body = @'
## 問題提起

## 学習者の仮説・考え

## Codexの整理

## 思考の変化

## 得た知見

## 次の確認問題
'@
gh issue comment 1 --body $body
```

新しいIssueを作る例:

```powershell
$body = @'
## 教科

## 学習テーマ

## 目的

## 現在の理解

## わからないこと

## まず自分で考えたこと

## 完了条件

- [ ] 自分の言葉で説明できる
- [ ] 類題または確認問題を1問以上解いた
- [ ] 間違いの原因を記録した
- [ ] 次回復習する内容を決めた
'@
gh issue create `
  --title "[学習] テーマ名" `
  --body $body `
  --label "状態:はじめられる" `
  --label "種類:しくみを知る" `
  --label "公開:のせる"
```

## 注意

- 学習者がまだ考えていない宿題やレポートは、完成物を代行せず、まず理解確認と構成作りにする。
- 出典が必要な内容は、根拠確認前に断定しない。
- 推測は「仮説」、未確認情報は「要確認」と書く。
- 記録は日本語で残す。
