# Auto Issue Recorder

会話中の疑問、問題提起、仮説、確認結果をIssueに記録してください。

## 方針

- 新しい学習テーマなら新しいIssueを作る。
- 進行中のテーマなら既存Issueにコメントする。
- 学習者の考えとCodex / Copilotの整理を分けて書く。
- 学習者が最初にどう考え、何をきっかけに見方を変え、どの知見を得たかを書く。
- 「なるほどポイント」と「次に使える判断基準」を残す。
- 未確認情報は「要確認」と書く。
- 類題または確認問題を1問以上用意する。
- 学習ログに残すべき区切りなら `learning-log/` の更新案も作る。
- 学習中のコメントでは、難しい評価用語を避け、小学生が答えやすい短い問いを添える。
- Issueコメントは、できるだけ `ふしぎ`、`予想`、`たしかめ`、`考え直し`、`なるほど`、`次のコツ` の一周として読めるように残す。
- 学習者が短く答えた場合も、勝手に考えを作らず、発言を材料にして「最初は何を考えたか」「どこを直したか」「次に何を見るか」を分けて記録する。
- `ハーネススタート`、`ハーネス開始`、または開始ニュアンスが明確な表現を受けた場合は、学習セッション開始として扱い、不足情報の確認、Issue化、学習用ブランチ作成に進む。
- ハーネス開始時は、Issue作成または最初のIssueコメントの前にラベルを整理する。最低でも `状態:*`、`種類:*`、`公開:*` を確認し、判断できる場合は `教科:*`、`助け:*`、`むずかしさ:*` も付ける。
- GitHub Pagesの学習レポートに載せる通常の学習は `公開:のせる` を付ける。個人情報、未整理の提出物、公開したくない内容は `公開:のせない` を付ける。`公開:のせる` と `公開:のせない` は同時に付けない。
- 学習用ブランチは `study/YYYYMMDD-topic-slug` または `study/YYYYMMDD-issue-N-topic-slug` の形式にする。
- `ハーネス終了`、`ハーネス完了`、または終了ニュアンスが明確な表現を受けた場合は、Close前チェックを行い、不足がなければ終了コメント、学習ログ、変更のコミットまたはPR、`main` 合流、Issue Close、次の課題候補まで整理する。
- `Portfolio` workflow はIssue closeイベントで `main` の内容を使うため、main合流後にIssueをCloseする。PR本文に `Closes #<Issue番号>` を入れる場合は、PR mergeによる自動Closeを終了処理の実行点とする。
- Close前チェックを満たさない場合は、Issueを閉じずに未完了タスクを示す。

## 開始時ラベル整理

`ハーネススタート`、`ハーネス開始`、または開始ニュアンスが明確な表現を受けたら、最初にIssueラベルを整理します。

Issueには、次の観点を付けます。

- 教科: 判断できる場合は `教科:*`
- 種別: 必ず1つの `種類:*`
- 状態: 必ず1つの `状態:*`。開始時は原則 `状態:はじめられる`
- 必要な支援: 学習モードに合う `助け:*`
- 難しさ: 判断できる場合は `むずかしさ:*`
- 公開可否: `公開:のせる` または `公開:のせない`

種別に迷う場合は、次の基準で仮決めします。

- `種類:しくみを知る`: 言葉や仕組みを理解する
- `種類:練習する`: 問題を解いて練習する
- `種類:まちがい発見`: 間違いを分析する
- `種類:ふりかえり`: 前に学んだことを復習する
- `種類:テスト練習`: テスト対策をする
- `種類:文章を書く`: 作文、レポート、記述を作る
- `種類:ふしぎメモ`: 疑問や論点を整理する

開始時一周記録には `## ラベル整理` を入れ、どのラベルを付けたか、または仮ラベルにした理由を残します。

## 開始時一周記録

`ハーネススタート`、`ハーネス開始`、または開始ニュアンスが明確な表現を受けたら、最初のIssue作成または最初のIssueコメントで、必ず「開始時一周記録」を残します。
これは思考ループの初回記録を欠かさないための必須処理です。

開始時一周記録では、同じIssueコメント内に次の3見出しを必ず入れます。

- `## 問題提起`
- `## 学習者の仮説・考え`
- `## 次の確認問題`

この3見出しは、思考ループ判定で `問い`、`仮説`、`確認` として読まれます。
見出し名を言い換えないでください。
短い開始コメントでも省略しないでください。

学習者の仮説がまだ不足している場合は、Codexが勝手に作らず、`## 学習者の仮説・考え` に「未確認。開始直後に本人の予想を確認する。」と明記します。
そのうえで、`## 次の確認問題` に「まず自分ではどう考えるかを1〜3文で書く」形式の問いを必ず置きます。

開始時一周記録の最小テンプレート:

```md
## 学習開始

ハーネス開始を受けて、このIssueを学習セッションの起点にする。

## ラベル整理

- 教科: `教科:*`
- 種別: `種類:*`
- 状態: `状態:はじめられる`
- 必要な支援: `助け:*`
- 公開可否: `公開:のせる` または `公開:のせない`

## 問題提起

学習者が今回わかりたいこと、解きたい問題、書きたいレポートテーマを書く。

## 学習者の仮説・考え

学習者がすでに書いた予想や考えを書く。
まだ出ていない場合は「未確認。開始直後に本人の予想を確認する。」と書く。

## Codexの整理

最初に分けて考える観点、確認する順番、注意点を短く整理する。

## 次の確認問題

まず自分ではどう考えるかを1〜3文で書く。

## 次回復習

未定。確認問題または最初の説明後に決める。
```

## コメント形式

```md
## 問題提起

## 学習者の仮説・考え

## 思考の変化

## なるほどポイント

## 得た知見

## 次に使える判断基準

## Codexの整理

## 根拠確認

## 誤解しやすい点

## 次の確認問題

## 次回復習
```

## 小学生向けの一周コメント

通常の見出しに加えて、Issueコメント内では次の短い聞き方を使えます。
終了時レポートは、この6段階をIssue本文とコメントから拾います。

```md
## ふしぎ

今いちばん知りたいことは何か。

## 予想

答えを見る前に、自分ではどう考えたか。

## たしかめ

何を見て、またはどの説明で確かめたか。

## 考え直し

確かめたあと、最初の考えからどこを直したか。

## なるほど

今回「なるほど」と思ったこと、説明できるようになったことは何か。

## 次のコツ

似た問題を見たとき、次は何から確認するか。
```

## 終了コメント形式

```md
## 学習終了まとめ

## 今日できるようになったこと

## 最初の見方

## 思考の変化

## 得た知見

## 次に使える判断基準

## 確認問題の結果

## 間違いの原因・誤解しやすい点

## 次回復習

## 次の課題候補
```
