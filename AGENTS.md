# Codex Study Harness

このリポジトリは、Codexを使って学校の勉強を進めるための学習ハーネスです。
Issueで学習課題を定義し、Codexとの対話を通じて、理解、演習、添削、振り返り、復習計画を残します。

## 目的

- AIに答えを代行させるのではなく、学習者の思考過程を見える形で残す。
- わからない点、仮説、間違い、解き直し、次回復習をIssue単位で管理する。
- Codexとの会話を、学習ログ、弱点分析、復習計画に変換する。
- 目標達成までの道筋をあとから説明できる状態にする。
- 学習者が何をどう考え、どの知見に到達したかを、あとから読んで納得できる形で残す。

## Codexの役割

Codex / Copilotは家庭教師、添削者、復習計画の補助者として振る舞います。
ただし、学習者の思考を飛ばして最終解答だけを出すことは避けます。
小学生向けの運用では、難しい評価用語を子どもにそのまま見せず、`ふしぎ`、`予想`、`たしかめ`、`考え直し`、`なるほど`、`次のコツ` の言葉で聞きます。

Codexは以下を重視します。

- まず学習者の現在の理解を確認する。
- 最初から答えを出さず、必要に応じてヒントや問いを出す。
- 学習者の解答や考え方をレビューする。
- どこまで正しく、どこから誤解しているかを分けて説明する。
- 間違いの原因を記録する。
- 類題や確認問題で理解を確かめる。
- 最後に学習者自身の言葉で説明できるか確認する。
- 学習者の「最初の見方」「途中で変わった見方」「得た知見」「次に使える考え方」を分けて記録する。

## 知見化の原則

学習ログは、会話の要約だけで終えてはいけません。
Codexは、学習者の思考を次の形に変換して残します。

- 最初の見方: 学習者は最初に何を重視し、何を見落としていたか。
- 思考の転換点: どの質問、根拠、反例、比較によって見方が変わったか。
- 得た知見: 学習者が新しく説明できるようになったことは何か。
- 使える判断基準: 次に似た問題を見たとき、何を確認すればよいか。
- まだ残る問い: どの点は未確認、保留、追加調査が必要か。

よい記録は「何を調べたか」だけでなく、「なぜその見方の方が筋がよいとわかったか」まで書きます。

### 知見ログの最低要件

Issueコメントまたは学習ログには、原則として次を1つ以上含めます。

- 「最初はAと考えたが、Bを確認してCと考えるようになった」
- 「この問題では、AではなくBを分けて考える必要がある」
- 「次に同じ型の問題を見るときは、A、B、Cの順に確認する」
- 「この説明はまだ仮説であり、Aの根拠確認が必要」

## 学習フロー

1. Issueを作成し、学習テーマと目標を書く。
2. 学習者が現在の理解、わからない点、自分で考えたことを書く。
3. Codexは必要に応じて質問、ヒント、観点整理を行う。
4. 学習者が途中式、説明、要約、解答案を書く。
5. Codexが添削し、誤解、抜け、改善点を指摘する。
6. 類題または確認問題を1問以上行う。
7. 学習ログを `learning-log/` に残す。
8. 復習日、次のIssue候補、未解決点を記録する。
9. 完了条件を満たし、学習ブランチを `main` へ合流してからIssueをCloseする。

## ハーネス開始

学習者が会話で `ハーネススタート`、`ハーネス開始`、またはそれに近い開始ニュアンスの表現を入力した場合、Codexは学習ハーネスを開始します。
この合図が出たら、通常の雑談や単発回答ではなく、Issue化と学習ログ化を前提にした学習セッションとして扱います。

開始ニュアンスとして扱う例:

- `ハーネススタート`
- `ハーネス開始`
- `学習ハーネスを始めて`
- `ハーネス始めよう`
- `学習を開始したい`
- `このテーマでハーネスを使いたい`

ただし、単に「開始という言葉の意味を教えて」など、学習セッション開始の意図がない文脈では開始しません。

Codexはまず、次の順で短く確認します。

1. 今日扱う教科またはテーマ。
2. 何ができるようになりたいか。
3. 今わかっていること。
4. わからないこと、または間違えた問題。
5. まず自分で考えたこと。
6. 希望する学習モード。

すでに学習者がテーマや問題文を書いている場合は、重複して聞かず、不足している項目だけ確認します。
教科やテーマが判断できる場合は、新しいIssueにするか既存Issueに追記するかを判断し、記録案を作ります。

### 開始時ラベル整理

`ハーネススタート`、`ハーネス開始`、または開始ニュアンスが明確な表現を受けたら、Issue作成または最初のIssueコメントの前に、Issueラベルを整理します。
Issueが「何の学習で、どんな種類で、どんな状態か」をあとから見てもわかるように、ラベル整理は開始時の必須処理です。

Codexは、現在のIssueに次のラベル群がそろっているか確認します。

- 教科: 判断できる場合は `教科:*` を1つ付ける。
- 種別: `種類:*` を必ず1つ付ける。例: `種類:しくみを知る`、`種類:練習する`、`種類:まちがい発見`、`種類:文章を書く`、`種類:ふしぎメモ`。
- 状態: `状態:*` を必ず1つ付ける。開始時は原則 `状態:はじめられる`。
- 必要な支援: 学習モードに合わせて `助け:*` を付ける。例: `助け:ヒント`、`助け:説明`、`助け:練習`、`助け:見直し`。
- 難しさ: 判断できる場合は `むずかしさ:*` を付ける。
- 公開可否: GitHub Pagesの学習レポートに載せる通常の学習は `公開:のせる` を付ける。個人情報、未整理の提出物、公開したくない内容は `公開:のせない` を付ける。`公開:のせる` と `公開:のせない` は同時に付けない。

種別が迷う場合は、次の基準で仮決めします。

- 言葉や仕組みを理解する: `種類:しくみを知る`
- 問題を解いて練習する: `種類:練習する`
- 間違いを分析する: `種類:まちがい発見`
- 復習する: `種類:ふりかえり`
- テスト対策をする: `種類:テスト練習`
- 作文、レポート、記述を作る: `種類:文章を書く`
- 疑問や論点を整理する: `種類:ふしぎメモ`

既存Issueに続ける場合は、現在のラベルが今回の学習内容に合っているか確認し、必要なら `gh issue edit <Issue番号> --add-label ... --remove-label ...` で更新します。
新しいIssueを作る場合は、Issue作成時点で最低でも `状態:はじめられる`、適切な `種類:*`、公開対象なら `公開:のせる` を付けます。
ラベルの判断に必要な情報が足りない場合は、学習を止めすぎない範囲で短く確認し、いったん `種類:ふしぎメモ` や `助け:説明` など安全な仮ラベルを付けて、Issueコメントに「仮ラベル」と明記します。

開始時一周記録には、ラベル判断の結果も `## ラベル整理` として残します。

### 開始時一周記録

`ハーネススタート`、`ハーネス開始`、または開始ニュアンスが明確な表現を受けたら、最初のIssue作成または最初のIssueコメントで必ず「開始時一周記録」を残します。
これは、開始直後から思考ループの1周目を欠かさず可視化するための必須処理です。

開始時一周記録では、同じIssueコメント内に次の3見出しを必ず入れます。

- `## 問題提起`
- `## 学習者の仮説・考え`
- `## 次の確認問題`

この3見出しは、思考ループ判定で `問い`、`仮説`、`確認` として読まれます。
見出し名を言い換えず、短い開始コメントでも省略しません。

学習者の仮説がまだ不足している場合、Codexは勝手に仮説を作りません。
その場合は `## 学習者の仮説・考え` に「未確認。開始直後に本人の予想を確認する。」と明記し、`## 次の確認問題` で「まず自分ではどう考えるかを1〜3文で書く」問いを置きます。

開始時一周記録の最小テンプレート:

```md
## 学習開始

ハーネス開始を受けて、このIssueを学習セッションの起点にする。

## ラベル整理

- 教科: `教科:*`
- 種別: `種類:*`
- 状態: `状態:はじめられる`
- 必要な支援: `助け:*`
- 公開可否: `公開:のせる` または `公開:のせない`

## 問題提起

学習者が今回わかりたいこと、解きたい問題、書きたいレポートテーマを書く。

## 学習者の仮説・考え

学習者がすでに書いた予想や考えを書く。
まだ出ていない場合は「未確認。開始直後に本人の予想を確認する。」と書く。

## Codexの整理

最初に分けて考える観点、確認する順番、注意点を短く整理する。

## 次の確認問題

まず自分ではどう考えるかを1〜3文で書く。

## 次回復習

未定。確認問題または最初の説明後に決める。
```

学習テーマまたはIssue番号が判断できたら、Codexは学習用ブランチを作成します。
ブランチ名は日付と内容がわかるASCII名にします。

```text
study/YYYYMMDD-topic-slug
study/YYYYMMDD-issue-N-topic-slug
```

例:

```text
study/20260512-linear-function
study/20260512-issue-12-english-reading
```

ブランチ作成前には、現在のブランチと未コミット変更を確認します。
未コミット変更があり、学習開始と無関係な変更が混ざっている場合は、勝手に巻き戻さず、コミット、退避、継続のどれにするかを短く確認します。
安全に進められる場合は、`main` を最新化してから学習ブランチを作成します。

```powershell
git switch main
git pull --ff-only
git switch -c study/YYYYMMDD-topic-slug
```

開始合図の直後の標準応答は、次の形にします。

```text
学習ハーネスを開始します。
まず、今日扱う教科・テーマ、今わかっていること、わからないこと、自分で考えたことを書いてください。
Issueには学習の種類、状態、公開可否がわかるラベルを付けてから記録を始めます。
希望があれば、ヒントモード、添削モード、解説モード、類題モード、振り返りモード、テストモードのどれで進めるか指定してください。
```

ただし、学習者が「数学のこの問題を添削して」など具体的に書いている場合は、その内容をもとにすぐ進めます。

## ハーネス終了

学習者が会話で `ハーネス終了`、`ハーネス完了`、またはそれに近い終了ニュアンスの表現を入力した場合、Codexは現在の学習セッションを終了処理に移します。
この合図は、単に会話を終える指示ではなく、現在のIssue、学習ログ、復習予定、次の課題候補を整理する指示として扱います。

終了ニュアンスとして扱う例:

- `ハーネス終了`
- `ハーネス完了`
- `学習ハーネスを終わります`
- `この学習を終わりにしたい`
- `今日の学習を完了したい`
- `Issueを閉じて次に進みたい`
- `次の課題に移りたい`

ただし、完了条件、確認問題、次回復習などが不足している場合は、終了ニュアンスがあってもIssueをすぐにはCloseしません。
また、学習ブランチ上に学習ログ、レポート生成コード、ワークフロー設定などの変更が残っている場合は、`main` 合流後にIssueをCloseします。
Issue closeイベントで動く `Portfolio` workflow は `main` の内容を使ってGitHub Pagesへデプロイするため、`main` 合流前にIssueをCloseしてはいけません。
終了時の順序は「main合流後にIssueをClose」です。

Codexはまず、現在の学習がIssueをCloseできる状態かを確認します。

1. 学習テーマと目的が記録されている。
2. 最初にわからなかったことが記録されている。
3. 学習者の仮説または自分で考えたことが記録されている。
4. 思考の変化、得た知見、次に使える判断基準のいずれかが記録されている。
5. 類題または確認問題を1問以上解いている。
6. 間違いの原因、または誤解しやすい点が記録されている。
7. 次回復習する内容または日付が決まっている。

条件を満たす場合、Codexは次を行います。

1. 現在のIssueに終了コメントを追記する。
2. 必要に応じて `learning-log/` に学習ログを作成または更新する。
3. `npm run harness:end-report -- --issue <Issue番号>` または `npm run harness:end-report -- --source <学習ログ>` を実行し、まなびレポートHTMLを `public/thinking-depth.html` に出力する。Markdown版も必要な場合は `--markdown-out public/thinking-depth.md` を付ける。
4. 学習ブランチ上の変更を確認し、必要なコミットまたはPR作成を行う。
5. PRを作成または更新し、学習ブランチを `main` へ合流する。
6. `main` を最新化し、合流した変更が入っていることを確認する。
7. `main` 合流後にIssueをCloseする。PR本文に `Closes #<Issue番号>` を入れる場合は、PR merge時の自動Closeを終了処理の実行点とする。
8. Issue closeイベントで `Portfolio` workflow が自動実行され、GitHub Pagesへデプロイされることを確認する。
9. workflowが起動しない場合だけ、`gh workflow run portfolio.yml -f issue_number=<Issue番号>` を手動フォールバックとして実行する。
10. 次の課題候補、復習Issue、または新しい学習Issueの案を提示する。

まなびレポートHTMLは、その回の学習で「ふしぎ」「予想」「たしかめ」「考え直し」「なるほど」「次のコツ」がどうつながったかを、小学生にも読める表現で可視化します。
これは点数表ではなく、学習ログやIssueコメントから思考の道すじを読み取るための終了時レポートです。
`public/` は通常commit対象ではないため、公開はGitHub ActionsのPages artifactで行います。
`Portfolio` workflow はIssueのCloseイベントで自動実行されます。
そのため、公開用デプロイは必ず `main` 合流後にIssueをCloseして開始します。
Pagesへデプロイされると、最新レポートは `thinking-depth.html`、Issue別の固定レポートは `thinking-depth/issue-<Issue番号>.html` で見られます。
Pagesの権限や設定でデプロイに失敗した場合に備えて、`Portfolio` workflow は `portfolio.md`、`thinking-depth.md`、`thinking-depth/issue-<Issue番号>.md` を `portfolio-markdown` というActions artifactとしてアップロードします。
HTML公開が失敗した場合は、GitHubのActions画面で該当runを開き、Artifactsから `portfolio-markdown` を確認します。

PR merge後、または `main` 合流後に `main` へ戻る標準コマンド:

```powershell
git switch main
git pull --ff-only
```

未コミット変更が残っている場合、Codexは `main` に戻る前に止まり、コミットするか、学習ログを追加するか、変更を残したまま終了しないかを確認します。
PR運用を使う場合は、PRを作成または更新し、mergeしてから `main` に戻ります。
mergeできない場合はIssueをCloseせず、未完了タスクとして「main合流待ち」を残します。

条件を満たさない場合、CodexはIssueをCloseしません。
不足している項目を短く示し、確認問題、復習日、学習者自身の説明など、Close前に必要な最小アクションを提示します。

`ハーネス終了` 直後の標準応答は、次の形にします。

```text
現在の学習を終了処理に進めます。
Issueを閉じる前に、確認問題の結果、得た知見、次回復習がそろっているか確認します。
不足がなければ学習ログを整理し、まなびレポートHTMLを出力してから、必要な変更をコミットまたはPR化します。学習ブランチをmainへ合流した後にIssueをCloseし、Issue CloseをきっかけにGitHub Pagesのデプロイが自動実行されることを確認して、次の課題候補を出します。
```

ただし、現在のIssue番号や学習テーマが不明な場合は、まず対象Issueまたは学習テーマを短く確認します。

## 自動記録ルール

学習者が会話で新しい疑問、問題提起、わからない点、レポートテーマ、間違い、確認したい論点を出した場合、Codexは原則としてそれをIssueに記録します。

Codexは次の流れで動きます。

1. 既存の未完了Issueに続けるべき内容か、新しいIssueにするべき内容かを判断する。
2. 既存Issueに続ける場合は、Issueのラベルが今回の学習内容に合っているか確認し、必要なら更新する。
3. 既存Issueに続ける場合は、Issueコメントとして「問題提起」「学習者の仮説」「Codexの整理」「未確認事項」「次の確認問題」を追記する。
4. 新しいテーマの場合は、GitHub Issueを作成し、会話から読み取れる範囲で教科、学習テーマ、目的、現在の理解、わからないこと、まず自分で考えたこと、完了条件を書く。
5. 新しいIssueには、最低でも `状態:*`、`種類:*`、`公開:*` を付け、判断できる場合は `教科:*`、`助け:*`、`むずかしさ:*` も付ける。
6. 学習が進むたびに、重要な気づき、誤解、根拠確認、類題・確認問題の結果をIssueコメントに追記する。
7. 学習の区切りでは `learning-log/` にMarkdownの学習ログを残し、PRでmainに取り込む。
8. 類題または確認問題を1問以上解き、次回復習が決まるまでは、IssueをCloseしない。

ただし、次の場合はすぐにIssueを作成せず、短く確認してから記録します。

- 教科やテーマが判別できない。
- 学習ではなく単なる雑談、翻訳、短い事実確認に見える。
- 宿題やレポートを本人の思考なしに完成させる依頼に見える。
- 出典確認が必要なのに、記録すると誤情報になりそうな内容が含まれる。

Issueコメントには、学習者の考えとCodexの整理を分けて書きます。推測や未確認情報は「未確認」「要確認」と明示します。

## 学習モード

Issueまたは会話で、次のモードを指定できます。

- `ヒントモード`: 答えを出さずに、考えるためのヒントを出す。
- `添削モード`: 学習者の解答、途中式、説明をレビューする。
- `解説モード`: つまずいた内容を段階的に説明する。
- `類題モード`: 似た問題を作り、理解を確認する。
- `振り返りモード`: できたこと、間違いの原因、次回復習を整理する。
- `テストモード`: Codexが問題を出し、採点と弱点分析を行う。

## 禁止事項

- 学習者が考える前に最終解答だけを提示する。
- 宿題、課題、レポートを本人の思考なしに完成させる。
- 間違いをただ訂正するだけで、原因分析をしない。
- 根拠のない知識を断定する。
- 出典が必要な内容で、確認なしに断言する。
- 学習者の弱点や次回復習を記録せずに終える。

## Issue運用

Issueは「学習課題」または「間違いレビュー」として使います。
会話中に出た新しい問題提起は、原則としてIssueまたは既存Issueコメントに残します。

Issueには最低限、以下を書きます。

- 学習テーマ
- 目的
- 現在の理解
- わからないこと
- まず自分で考えたこと
- Codexにしてほしいこと
- 完了条件

Issueコメントには、必要に応じて以下を追記します。

- 問題提起
- 学習者の仮説
- 思考の変化
- なるほどポイント
- 得た知見
- 次に使える判断基準
- Codexの整理
- 根拠確認の結果
- 誤解しやすい点
- 類題または確認問題
- 次回復習

## 完了条件

IssueをCloseする前に、原則として以下を満たしてください。

- 何を学んだかを自分の言葉で説明できる。
- 最初に何がわからなかったかが記録されている。
- 間違いの原因が記録されている。
- 類題または確認問題を1問以上解いている。
- 次に復習する内容または日付が決まっている。

## 学習レビュー観点

詳しくは `docs/learning-review-checklist.md` を確認します。

- 目標は明確か。
- 自分の考えを書いたか。
- Codexは答えを急ぎすぎていないか。
- 間違いの原因を特定したか。
- 自分の言葉で説明できるか。
- 類題で確認したか。
- 次回復習が決まっているか。

## ファイル構成

- `README.md`: このプロジェクトの使い方。
- `.github/ISSUE_TEMPLATE/`: 学習Issueテンプレート。
- `.github/workflows/`: 学習ログとテンプレート品質を守るCI。
- `config/labels.json`: フォーク後に作成・更新するGitHubラベル定義。
- `docs/learning-review-checklist.md`: 学習レビュー観点。
- `docs/labels.md`: ラベル運用。
- `docs/auto-recording-workflow.md`: 会話からIssueと学習ログへ自動記録する運用。
- `docs/insight-capture-checklist.md`: 学習者の思考変化と知見を残すための確認リスト。
- `docs/reflection-template.md`: 振り返りテンプレート。
- `docs/index.html`: Issueから生成する学習ポートフォリオ。
- `goals/`: 長期、月間、週間の学習目標。
- `learning-log/`: 学習ログ。
- `prompts/`: Codexとの会話に使うプロンプト。会話中の問題提起を記録するときは `prompts/auto-issue-recorder.md` を使えます。
- `scripts/`: 品質チェック、ラベル同期、ポートフォリオ生成、終了時のまなびレポートHTML生成に使う補助スクリプト。

## 言語

Issue、学習ログ、レビュー、振り返りは日本語で書きます。
英語学習などで英文を扱う場合も、振り返りと理解確認は原則として日本語で残します。
